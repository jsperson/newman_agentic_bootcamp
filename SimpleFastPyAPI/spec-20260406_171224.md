# Codebase Spec: SimpleFastPyAPI

## What does this project do?
A simple REST API built with Python, FastAPI, and SQLAlchemy that provides CRUD operations (Create, Read, Update, Delete) on a `users` resource backed by a SQLite database.

## Tech Stack
- Language: Python 3.10 (Docker), system Python 3.12 (local `.pycache` suffixes)
- Framework: FastAPI 0.109.1, SQLAlchemy 1.4.47, Pydantic 1.10.13
- Key dependencies/libraries (with versions):
  - `fastapi==0.109.1` — web framework
  - `starlette==0.49.1` — ASGI foundation (⚠ incompatible with FastAPI 0.109.1)
  - `SQLAlchemy==1.4.47` — ORM (v2 is current)
  - `pydantic==1.10.13` — data validation (v2 is current)
  - `uvicorn==0.21.1` — ASGI server
  - `psycopg2-binary==2.9.6` — PostgreSQL driver (unused; project uses SQLite)
  - `python-dotenv==1.0.0` — env var loading (imported nowhere)

## Architecture
- Single entry point `main.py` defines the FastAPI app and all 5 route handlers.
- `app/database.py` holds a hardcoded SQLite connection string, engine, session factory, and a `get_db` dependency.
- `app/models.py` defines a single SQLAlchemy declarative model `User`.
- `app/schema.py` defines two Pydantic schemas (`UserCreate`, `UserUpdate`) for request validation.
- No service layer, no repository pattern, no middleware, no lifespan/startup hooks.
- Database tables are never created programmatically — `Base.metadata.create_all()` is never called.
- The `.env.example` file references PostgreSQL config, but the code hardcodes SQLite — the two are disconnected.

## Issues Found
| # | Issue | Severity | File(s) | Details |
|---|-------|----------|---------|---------|
| 1 | **No programmatic table creation — relies on committed DB file** | High | `app/database.py` (all lines) | `Base.metadata.create_all(bind=engine)` is never called. The app only works because `users.db` is committed to the repo with the table pre-created (3 seed rows: Alice, Bob, Carol). If the DB file is missing or deleted, the app crashes with `sqlite3.OperationalError: no such table: users`. |
| 2 | **Passwords stored and returned in plain text** | Critical | `main.py:26`, `app/models.py:11`, README lines 62–83 | No hashing (bcrypt/argon2). POST stores raw password; all GET responses expose it. README examples show `"password": "password1"` in JSON output. |
| 3 | **Starlette version incompatible with FastAPI** | Critical | `requirements.txt:29` | `starlette==0.49.1` requires FastAPI ≥ 0.115.0. FastAPI 0.109.1 pins `starlette>=0.36.3,<0.37.0`. `pip install` will fail or produce undefined behavior. |
| 4 | **Pydantic v1 with outdated starlette** | High | `requirements.txt:21,29` | Pydantic 1.10.13 is EOL. FastAPI 0.109.1 works with it, but the starlette mismatch above makes the whole install broken. |
| 5 | **Function names lie — filter by ID, named "by_email"** | High | `main.py:17,33,43` | `get_user_by_email`, `update_user_by_email`, `delete_user_by_email` all filter on `User.id == user_id`. Names should be `get_user`, `update_user`, `delete_user`. |
| 6 | **No duplicate-email handling on POST** | High | `main.py:24-30` | `email` is `unique=True` in `app/models.py:10`, but `create_user` doesn't catch `IntegrityError`. A duplicate POST returns a 500 instead of a 409/422. |
| 7 | **`UserUpdate` requires all fields (no partial update)** | High | `app/schema.py:9-11` | Both `name` and `email` are required. To change only the name you must resend the email. Fields should be `Optional[str] = None`. |
| 8 | **PUT returns a message, not the updated resource** | Medium | `main.py:40` | Returns `{"message": "User updated successfully"}` instead of the updated user object. Inconsistent with POST which returns the created user. |
| 9 | **No `response_model` on any endpoint** | Medium | `main.py` (all routes) | Without `response_model`, Pydantic doesn't strip sensitive fields (passwords leak). Also loses automatic OpenAPI response schema generation. |
| 10 | **Database config mismatch: env vs code** | Medium | `.env.example` (all lines), `app/database.py:4` | `.env.example` defines PostgreSQL vars (`DB_HOST`, `DB_PORT=5432`, etc.), but `database.py` hardcodes `sqlite:///./users.db`. `python-dotenv` is in requirements but never imported. |
| 11 | **`psycopg2-binary` is an unused dependency** | Medium | `requirements.txt:20` | PostgreSQL driver is listed but the project uses SQLite. Adds unnecessary install time and image size. |
| 12 | **`users.db` committed to git** | Medium | `.gitignore`, `users.db` | The SQLite file is tracked in the repo. It should be in `.gitignore`. An empty DB file is also useless — schema creation should happen at runtime. |
| 13 | **`--reload` in production Docker CMD** | Medium | `Dockerfile:8` | `--reload` is for development only. It adds overhead and can cause issues in production. |
| 14 | **Docker container runs as root** | Medium | `Dockerfile` (all lines) | No `USER` directive. The container runs as root, which is a security risk. |
| 15 | **DELETE endpoint curl example is wrong in README** | Low | `README.md:163` | `curl -X DELETE http://localhost:8000/2` — missing `/users/` prefix. Should be `/users/2`. |
| 16 | **`typing==3.7.4.3` is obsolete** | Low | `requirements.txt:30` | The `typing` backport is only needed for Python < 3.5. Python 3.10+ has it in stdlib. |
| 17 | **SQLAlchemy type annotations are misleading** | Low | `app/models.py:8-11` | `id: int = Column(...)` suggests the attribute is an `int`, but at class-definition time it's a `Column` descriptor. Should use `Mapped[int]` (SQLAlchemy 2.0) or drop the annotations. |
| 18 | **No input validation on email format** | Low | `app/schema.py:6` | `email: str` accepts any string. Should use `EmailStr` from `pydantic` or `email-validator` (which is already in requirements). |
| 19 | **No pagination on GET `/users/`** | Low | `main.py:12-14` | `db.query(User).all()` loads every row into memory. Will break at scale. |
| 20 | **No tests** | Low | (entire project) | Zero test files. No `pytest`, no `httpx` test client usage. |
| 21 | **`python-dotenv` never used** | Low | `requirements.txt:22`, no imports | Listed in requirements but never imported or called anywhere. |
| 22 | **`app/__init__.py` is empty** | Trivial | `app/__init__.py` | Could at least expose `Base`, `engine`, or version info. |

## Notes
- The `users.db` SQLite file exists on disk but contains zero tables (confirmed via `sqlite3 .tables` returning empty). This corroborates Issue #1 — tables were never created.
- The `.pycache` directories show `cpython-312` suffixes, meaning someone ran this on Python 3.12 locally, while the Dockerfile targets Python 3.10.
- The project has 5 endpoints covering full CRUD but lacks authentication, authorization, rate limiting, error handling middleware, logging, and health checks.
- README installation instructions are out of order (Docker build/run listed before `pip install`).
